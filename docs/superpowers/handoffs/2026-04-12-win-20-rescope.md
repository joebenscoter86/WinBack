## What's broken (must rescope)

### Problem 1: the evidence mapper is structurally wrong

Current mapper (`backend/lib/disputes/build-evidence-payload.ts`) assumes each evidence file can be assigned 1:1 to a Stripe evidence slot, and uses a "collision handler" to drop extras.

**Reality:** Stripe's dispute evidence API has exactly 9 file slots (`receipt`, `customer_communication`, `customer_signature`, `shipping_documentation`, `service_documentation`, `cancellation_policy`, `refund_policy`, `duplicate_charge_documentation`, `uncategorized_file`). Each slot accepts exactly **one** `file_id`, not an array. Real merchants submit 10+ files all the time — Stripe's own dashboard handles this by **concatenating multiple uploaded files into a single PDF per slot** before the update call.

**QA proof:** On du_1TLbFcEQYvM3XwRzuHQaXOj1 (warning_needs_response, visa 10.4), user uploaded 4 files to 4 different checklist items. Three of those items mapped to `uncategorized_file` in the playbook. The mapper put one in `uncategorized_file`, dropped two with `field_collision` warnings, and the Stripe submission succeeded with only **2 of 4 files attached**. The AI narrative referenced the dropped files by name, so an issuer would see prose pointing at evidence that isn't there. Catastrophic correctness bug. Submission DB row id: `ca7bda0f-79ce-46a0-9bf5-c437f4cc4d8b`.

**The right answer is PDF concatenation at submit time.** Roughly:

1. Group evidence files by their target slot (`stripe_evidence_field` on the playbook item, or narrative-text for text references)
2. For any slot with >1 file, download each via Stripe Files API, use `pdf-lib` (images become pages, PDFs get page-appended, unknown types go to `uncategorized_file` as-is), upload the combined PDF back to Stripe Files with `purpose: "dispute_evidence"`, get one new file_id
3. For slots with exactly 1 file, pass through unchanged
4. Assign the (possibly combined) single file_id to the slot
5. Submit

`buildEvidencePayload` becomes async, takes a Stripe client as a dependency, and loses the `field_collision` + `dropped` warning codes. Supabase Storage might be useful as a staging area for large bundles, or the whole thing can run in memory depending on file sizes. Think about size limits — Stripe's dispute evidence per-file limit and per-submission limit both matter.

### Problem 2: the playbook `stripe_evidence_field` assignments are sloppy

Task 3 in the current plan told me "when in doubt, `uncategorized_file`." Terrible guidance — I meant "when I can't decide" but it wrote the bug into 7 playbook files. Every playbook now has multiple items mapping to `uncategorized_file` as a lazy default.

Even after PDF concat fixes the mechanical "files dropped" bug, the playbook semantics need a second pass:

- **Some items should not accept files at all.** "IP address or device ID/fingerprint matching" is *data to cite in the narrative*, not a document. These should be marked as narrative-only items in the `EvidenceChecklistItem` schema (add a `narrative_only: true` flag or similar), and the UI should not show a file uploader for them.
- **Genuinely file-bearing items should have semantically correct slot assignments.** "Bank statement name screenshot" is a screenshot the cardholder would see — `customer_communication`? `uncategorized_file`? Needs a thoughtful pass per playbook. Joe has 10+ years of issuer-side experience; if you're unsure, ask.

Playbook data files: `backend/lib/playbooks/data/{visa-10.4,visa-13.1,visa-13.2,visa-13.3,visa-13.6,mastercard-4808,mastercard-4853}.ts`.

**Gotcha:** playbooks are loaded at runtime from the Supabase `playbooks` table via `getPlaybook()`, not from the TS files directly. Any playbook edit requires `cd backend && set -a && source .env.local && set +a && npm run seed:playbooks`. This trap already cost a full QA cycle — see the relevant section in [CLAUDE.md](../../CLAUDE.md).

### Problem 3: merchants don't see warnings on success

Current `SubmissionConfirmation.tsx` just says "submitted!" — it does not surface the `warnings` array from the backend response. Even after PDF concat lands there will still be legitimate warnings to show (e.g., `missing_mandatory_items`, `field_truncated`, future warnings from the concat step if files are dropped for size). Design for this — the success screen should be able to render a warnings list clearly.
