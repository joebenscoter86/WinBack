## The four items

### Bug 1 — Concat broken on PDFs (MUST FIX)

**Symptom:** The WIN-20 multi-file PDF concat path silently never runs when the slot contains any PDF. On the QA submit, service_documentation was expected to produce a combined PDF of `IMG_5417.jpeg + invoice-004-international.pdf`, but instead the PDF got skipped with a `concat_skipped` warning and the slot fell through to single-file pass-through of just the JPEG. `concat_receipts` on the submissions row is null. **The multi-file merge codepath has never been exercised end-to-end in live QA.**

**Root cause:** Two problems stacked:

1. **Assembler mime check is broken.** `backend/lib/disputes/assemble-evidence.ts` contains:
   ```ts
   const kind = file.mime_type === "application/pdf" ? "pdf" : "image";
   ```
   The `evidence_files.mime_type` column stores **extensions** (`"pdf"`, `"jpg"`, `"png"`), not full MIME types. So the comparison never matches, PDFs route through the image branch, and `sharp` rejects them with `"Input buffer contains unsupported image format"`.

2. **The frontend upload code is the reason the column has the wrong shape.** Somewhere in the stripe-app upload path, when persisting to `evidence_files` the code saves the extension instead of `fileObject.type` (or whatever the real MIME is). The DB column's original contract (per the plan and spec) is that it holds full MIME types. That contract got violated.

**Fix: do both, in this order.**

**1a. Defensive assembler fix (ships correctness immediately).** In `backend/lib/disputes/assemble-evidence.ts`, replace the `mime_type === "application/pdf"` check with something that accepts all the forms we'll realistically see:
```ts
function isPdfFile(file: EvidenceFileInput): boolean {
  const mime = file.mime_type?.toLowerCase() ?? "";
  if (mime === "application/pdf" || mime === "pdf") return true;
  if (file.file_name?.toLowerCase().endsWith(".pdf")) return true;
  return false;
}
// ...
const kind = isPdfFile(file) ? "pdf" : "image";
```
Add a unit test to `backend/lib/disputes/assemble-evidence.test.ts` that verifies a file with `mime_type: "pdf"` is treated as a PDF (use the existing `sample.pdf` fixture in `backend/__tests__/fixtures/concat/`).

**1b. Root cause — frontend upload MIME type.** Grep for where `evidence_files` rows get created with their `mime_type`. Suspects:
- `stripe-app/src/components/evidence/FileUploadSection.tsx` — `handleUploadComplete` builds the POST body
- `backend/app/api/disputes/[disputeId]/evidence-files/route.ts` — the API route that inserts the row

One of those two is mapping `fileObject.type` to an extension. Find it, restore the MIME type, and while you're there decide whether to backfill the existing 16 rows in dev Supabase. Backfill is easy — a one-shot SQL UPDATE that maps `mime_type = 'pdf'` → `'application/pdf'`, `'jpg'`|`'jpeg'` → `'image/jpeg'`, `'png'` → `'image/png'`. Not strictly required (the defensive fix in 1a handles existing rows), but it's the clean move.

**Verification:** The re-QA run should produce a multi-file slot that actually concats (upload 3 PDFs to a single slot, verify `concat_receipts` populates with the combined file_id and the combined PDF opens in the Stripe Dashboard showing all 3 pages).

### Bug 2 — Frontend autofill renderer gap (MUST FIX)

**Symptom:** During QA, the merchant uploaded `invoice-006-legal.pdf` to the "Bank statement name screenshot" item on visa-10.4. That item is **Category A** in the spec matrix with `stripe_field: "calculated_statement_descriptor"`. The file should have never had an uploader rendered for it, and if one did render and the merchant used it, the file should at least have reached Stripe. Instead:
1. The upload UI rendered anyway (frontend renderer gap)
2. The file was saved to `evidence_files`
3. The assembler silently dropped it because the item has no `stripe_evidence_field`
4. The merchant has no idea the file went into the void

This is **silent data loss** — exactly the class of bug the rescope was supposed to eliminate.

**Root cause:** `stripe-app/src/components/evidence/ChecklistItem.tsx` (or wherever the WIN-19 `stripeFieldResult` pattern lives — grep for that identifier) only knows how to show the autofilled hint for the WIN-19 set of stripe_field values: `authorization`, `avs_result`, `cvc_check`, `three_d_secure`. WIN-20 added three more values to the playbook data — `calculated_statement_descriptor`, `customer_email`, `refund_data` — but the renderer never got updated to recognize them. Those items fall through to the default upload UI.

I confirmed the DB is correct: `visa-10.4` item #12 has `stripe_field: "calculated_statement_descriptor"` in the live Supabase `playbooks` row. So the bug is frontend-only.

**Fix:** In the autofill renderer, branch on `item.stripe_field` being **truthy** (hide the uploader, show the "We already have this from Stripe" hint) instead of branching on a hardcoded set of known values. Don't gate on the specific `stripe_field` value unless you need to customize the hint copy per field — even then, fall back to a generic "Autofilled from Stripe" label for unknown values rather than the upload UI.

You may also need to update `backend/lib/disputes/assemble-evidence.ts` `writeAutofilledEvidenceField` to handle `calculated_statement_descriptor` properly. Per Task 7's implementation, calculated_statement_descriptor currently lands via narrative template substitution rather than a dedicated evidence text field — that's fine from a Stripe submission standpoint, but worth re-reading the switch to confirm the value actually flows through.

**Verification:** Re-QA: look at visa-10.4 and confirm items #1, #2, #3, #7, #12 all show the autofilled hint and have NO upload UI. Items #5, #6, #11 still show "Covered in your narrative" (unchanged). Items #4, #8, #9, #10, #13 still show the upload UI.

### Bug 3 — Submit screen "N of M attached" counter (MUST FIX)

**Symptom:** Submit screen shows "2 of 8 attached" for mandatory evidence even when the merchant has attached files to every uploadable mandatory slot AND 3+ items are autofilled. This is merchant-visible wrongness.

**Root cause (likely):** The counter iterates over `mandatory` items and checks for a matching `evidence_files` row, without excluding items where `stripe_field` is set (autofilled, implicitly satisfied) or `narrative_only` is true (implicitly satisfied). The backend guard at `backend/lib/disputes/submission-guard.ts` was updated in Task 8 to filter A + T items from the missing-mandatory check, but the frontend counter is a separate computation that was never updated.

**Fix:** Grep the stripe-app for whatever computes the "X of Y attached" string (likely `filter(i => i.category === 'mandatory')` or similar). Mirror the Task 8 guard logic:
```ts
const relevantMandatory = playbook.evidence_checklist.filter((i) =>
  i.category === 'mandatory' && !i.narrative_only && !i.stripe_field
);
const attached = relevantMandatory.filter((i) => hasFile(i.item));
// "attached.length of relevantMandatory.length attached"
```

**Verification:** On a fresh visa-10.4 dispute with all 5 slot items attached, the counter should show "5 of 5". With 3 attached and 2 missing, it should show "3 of 5".
