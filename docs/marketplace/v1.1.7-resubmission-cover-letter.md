# v1.1.7 Resubmission - Cover Letter for Stripe Marketplace Review

**Submit with:** v1.1.7 marketplace resubmission
**Addresses:** v1.1.6 rejection - "Test/Live mode data isolation"
**Date prepared:** 2026-05-04

---

Hi Stripe Review Team,

Thank you for the v1.1.6 rejection feedback. The issue was real and we have fixed it. This cover letter explains what we changed, how we verified it, and the architectural decisions behind the fix.

## What you saw in v1.1.6

When you toggled the merchant's dashboard to **Live mode** and opened WinBack, you saw the test-mode disputes from the merchant's account instead of the (empty) live-mode list. You correctly flagged this as a data isolation regression.

## Root cause

WinBack's backend used a single boot-time `STRIPE_SECRET_KEY` environment variable to construct one Stripe API client at process start. That key was a `sk_test_*` key, so every dispute the backend fetched - regardless of which dashboard mode the merchant was in - came from the test-mode Stripe API. The frontend had no way to influence this; the backend had no concept of per-request mode.

## What v1.1.7 changes

We refactored to **per-request mode selection**. Specifically:

1. **Frontend now sends mode on every request.** The Stripe Apps SDK exposes `context.environment.mode` (`'live'` or `'test'`). Our `apiClient.ts` reads that and includes `livemode: boolean` in every backend request body.
2. **Backend trusts the signed body's `livemode` claim.** After `STRIPE_APP_SECRET` HMAC verification passes, the backend reads `livemode` from the body and uses it to select between two new env vars: `STRIPE_SECRET_KEY_LIVE` and `STRIPE_SECRET_KEY_TEST`. A mode-aware Stripe client (`lib/stripe/client.ts`) caches one Stripe instance per mode.
3. **Every Supabase query filters by mode.** We added a `livemode boolean` column to `disputes`, `dispute_submissions`, `narrative_generations`, `evidence_files`, and `webhook_events`. Every read filters by `.eq("livemode", livemode)` and every write persists the mode.
4. **Webhook handler verifies against both webhook secrets.** Live secret first, test secret as fallback, 400 if both fail. The persisted `event.livemode` is written to every `webhook_events` row and every dispute upsert/update.
5. **Cron reconciliation runs once per merchant per mode.** The reconcile job iterates `[true, false]` for each merchant so neither mode gets stale.

## Architectural note on Stripe App signing secret

Our initial fix attempted to use mode-suffixed `STRIPE_APP_SECRET_LIVE/_TEST` env vars (parallel to the API keys). We confirmed empirically that **Stripe Apps issues exactly one signing secret per app, regardless of dashboard mode**. The same `absec_*` value appears in the Stripe app dashboard whether the toggle is set to live or test. So we kept a single `STRIPE_APP_SECRET` and used the body's `livemode` claim (covered by the signature via the body content) as the mode source. The signature still proves the request originated from a legitimate Stripe iframe; the mode field can only be falsified by a merchant against their own account, where they already have access to both modes.

## How we verified the fix

- **494 unit tests** covering signature verification, mode-aware client, every route's mode plumbing, and webhook handler try-both fallback.
- **10 integration tests** that hit a real Supabase dev DB with mocked Stripe and Anthropic. Includes a dedicated `livemode-isolation.test.ts` that fires both a live-mode and a test-mode signed request at `/api/disputes` and asserts each only returns its own mode's data, and that the `is_new` flag query filters by mode.
- **Live deployment smoke test.** `backend/scripts/smoke-livemode.ts` hand-signs four requests against the production runtime: valid+test, valid+live, forged signature, and missing-livemode-claim. The first two return 200 with mode-isolated data, the latter two return 401. We ran this against the v1.1.7 production deployment after merge.

## A note on live-mode QA

We attempted to install v1.1.7 in live mode against a Stripe account in our publisher org (JB Technology LLC) to reproduce your original repro path. The Stripe dashboard rejected the install with: *"You can't install an app that's been published on this account."* This appears to be an org-level publisher restriction. We emailed the reviewing engineer asking for a recommended QA path; we didn't receive a reply before submission.

We compensated for the missing live-install QA with the smoke script above, which exercises the full deployed runtime in both modes via signed requests. The smoke output is in the next section.

## Smoke output

Run against `https://winbackpay.com` on 2026-05-05:

```
Smoke target: https://winbackpay.com

[t1] valid sig + livemode=false: 200 data.length=4
[t2] valid sig + livemode=true:  200 data.length=0
[t1+t2] mode isolation: t1=4 test, t2=0 live, overlap=0
[t3] forged secret:               401
[t4] missing livemode:            401

Smoke passed.
```

A test-mode signed request returns 4 test disputes from this account. A live-mode signed request returns zero (the account has no live disputes, as expected). The ID set overlap is empty - no dispute appears in both modes. Forged signatures and missing-livemode payloads are rejected at the auth layer with 401.

## What's the same

- `STRIPE_APP_SECRET` (singular) verifies all signed requests.
- The frontend permission set, UI, and post-install action are unchanged from v1.1.6.
- The backend URL (`https://winbackpay.com/api/`) and CSP are unchanged.

## Reproduction steps for your reviewer

1. Install WinBack v1.1.7 on a Stripe account that has at least one test-mode dispute (use `stripe trigger charge.dispute.created` against an account with `--api-key` set to that account's test secret).
2. Open the WinBack drawer. You should see the test dispute(s).
3. Toggle the dashboard to **Live mode**. The drawer should now show **only** live-mode disputes for that account (likely zero unless you have real disputes, in which case those and only those).
4. Toggle back to **Test mode**. Test disputes reappear.

We are confident this fully addresses the v1.1.6 rejection. Please let us know if you find anything else.
